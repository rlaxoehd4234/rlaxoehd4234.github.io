## 1️⃣ WAS + JAR 구조 이해하기

### ⚙️ WAS(Tomcat)의 역할
- 클라이언트 요청 수신 (포트 리스닝)
- JSP/Servlet 실행 환경 제공
- 스레드 관리 및 요청 분배
- Front Controller 역할

### 📦 JAR 모듈의 역할
- 비즈니스 로직 처리
- DB 접근, 트랜잭션 관리
- 별도 로그 기록 및 유틸 기능 제공

### 🔄 실행 흐름
Client → WAS(Tomcat JSP/Servlet) → JAR(비즈니스 로직) → DB → 결과 반환

- JAR은 “실제 로직과 데이터 처리 담당”

## 2️⃣ 물리 서버 vs 클라우드 서버 비교

| 구분 | 물리 서버 | 클라우드 서버 |
|------|-----------|---------------|
| **구축 비용** | 초기 장비 구입 비용 큼 | 초기 비용 적고 사용량 기반 과금 |
| **운영** | 하드웨어/전원/네트워크 직접 관리 필요 | 인프라 제공 업체가 기본 관리 (AWS, GCP, Azure 등) |
| **확장성** | 인프라 확장에 시간과 비용 많이 소요됨 | 필요 시 빠른 스케일 업/아웃 가능 |
| **장애 대응** | HW 장애 시 직접 대응해야 함 | 자동 백업 / 장애 복구 옵션 있음 |
| **보안 제어** | 내부망 완전 통제 가능 | 보안 그룹, 네트워크 ACL 등 설정 필요 |
| **사용 사례** | 공공기관, 내부망 전용 시스템 | 외부 서비스, 유연성과 확장 필요 시스템 |

---

## 3️⃣ 보안 고려사항

### 🔒 네트워크 보안
- 외부망/내부망 분리 (DMZ 구성)
- 불필요한 포트 차단 (Tomcat 포트, DB 포트 등)
- 클라우드 환경 시 보안 그룹 / VPC / 서브넷 설정 강화

### 🔑 인증 & 암호화
- GPKI 또는 SSL 인증서 사용
- 파일 손상 방지 및 전송 시 암호화
- DB 접근 계정과 비밀번호 안전하게 관리 (환경변수/암호화 vault)

### 📝 로그 / 개인정보 보호
- 주민번호, 차량번호 등 민감정보 마스킹 또는 암호화
- 로그 접근 권한 최소화
- 로그 보관 정책 및 주기적 보관 ≠ 무제한

### ⚙️ 운영 측면
- 최소 권한 원칙 (root/admin 계정 사용 최소)
- 정기적인 보안 패치 / 업데이트
- Scheduler 중복 실행 방지 (Lock 처리)
- 장애 대응 및 백업 정책 마련
- 모니터링 / 알림 시스템 구성 (CPU, 메모리, 스레드, DB 연결 상태 등)

# ✅ 결론

- **WAS + JAR 구조**: 역할을 명확히 분리해서 유지보수, 확장성 확보  
- **물리 vs 클라우드 서버**: 운영 환경 특성에 따라 선택  
- **보안**: 방화벽, 인증, 로그, 운영 정책 등 기본부터 철저히  

> 최종 운영 환경을 준비할 때는 이 모든 요소가 고르게 고려되어야 합니다. 준비된 환경만이 안정적이고 신뢰성 있는 시스템을 만들어낼 수 있어요.
